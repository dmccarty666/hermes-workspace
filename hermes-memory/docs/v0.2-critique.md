## 2. Top-Severity Issues (resolve before Sprint 1)

### Issue 1: `agent_ref` Injection Plan Is Hopeful, Not Proven

**Severity:** **High**
**Category:** Architecture risk — narrative-thread fix
**Refs:** TDD §9.2, §15.1; Plan Story 5.1.2

The v0.2 fix for `/new` injection requires the plugin to insert a user message into `AIAgent.conversation_history` via a stored reference (`agent_ref`). The TDD acknowledges the existing `MemoryProvider.initialize(session_id, **kwargs)` does NOT currently pass `agent_ref`. The plan says "try reflection first, raise an upstream PR if needed."

**Problems:**

1. The reflection escape hatch is brittle. Hermes' `AIAgent` internal structure can change. Plugin breaking silently is the worst outcome — narrative-thread fails closed (no injection), but tests would need to actually trigger `/new` to catch it.
2. The upstream PR path adds calendar time we can't predict — could be days or weeks before it merges, blocking Phase 5 acceptance.
3. There's actually a **third option** that's cleaner and we missed: use `MemoryManager._agent` if Hermes' manager exposes it (need to check), OR ride on `on_pre_compress` style hooks where Hermes does pass message context. Worth a 30-min spike to enumerate ALL access paths before betting on `kwargs.get('agent_ref')`.

**Recommendation:**

Spike before Sprint 5: enumerate every available path the plugin can use to mutate or prepend to conversation_history. Document them in an ADR under `docs/adr/001-narrative-thread-injection.md` with pros/cons. **Do not start Phase 5 implementation until this is settled.** If the only viable path is an upstream Hermes PR, file that PR at the start of Phase 1 so it has time to merge.

### Issue 2: Plugin/Gateway Concurrent Writes — SQLite Advisory Lock Is Hand-Waved

**Severity:** **High**
**Category:** Data integrity
**Refs:** TDD §6.4, §14.1

TDD §6.4 says the plugin and gateway "coordinate via in-DB advisory locks for dreamer batch writes." This is mentioned in one sentence and then never specified.

**Problems:**

1. SQLite has no native advisory locks. The TDD's reference to "in-DB advisory lock as a row in audit_log" is hand-wavy. Two processes can both insert a marker row without coordination — that's not a lock, that's a record.
2. The actual contention scenarios are not enumerated:
   - Plugin writes `sync_turn` → INSERT into turns
   - Gateway (dreamer) bulk UPDATE on turns.dream_status
   - Both could deadlock on the FTS5 triggers depending on busy_timeout
3. WAL mode allows concurrent reads but **only one writer at a time across processes**. If the dreamer is mid-run and the user is using the CLI, the CLI's `sync_turn` insert will block (`SQLITE_BUSY`) — and the plugin has no explicit timeout/retry policy documented.

**Recommendation:**

- Adopt one of these explicit strategies before Sprint 1:
  - **(A) Process-pinned writes:** Plugin is the only writer to certain tables (sessions, turns, raw_events). Gateway+dreamer is the only writer to others (facts, decisions, dream_runs). They can't conflict because they don't touch the same tables. Reads are free for both.
  - **(B) Single-writer queue:** Both plugin and gateway send write requests through a single in-process writer thread (whoever holds the connection); the other process posts to a Unix socket / named pipe queue.
  - **(C) Accept the contention:** Set `busy_timeout=30000` (30s); document that during the 3am cron window the CLI may briefly stall.

Pick (A) for MVP. It's the simplest and the table separation is natural.

- Spell out the write-ownership table in TDD §6.4.
- Add retry-with-backoff to the plugin's SQLite calls explicitly.

### Issue 3: Indexer Runs as a "Background Worker" — Where Does It Actually Live?

**Severity:** **High**
**Category:** Process model
**Refs:** TDD §4.2, Plan Story 3.2.2

The async indexer (chunk → embed → upsert Qdrant) is described as "a background worker thread (or systemd-style)." That's two very different architectures.

**Problems:**

1. If it's an **in-process thread inside Hermes**, the indexer dies when the CLI quits. A user who runs a 200-turn session, then quits, may leave 50+ turns un-indexed forever — they won't appear in semantic search.
2. If it's a **separate systemd service**, it can run anytime — but then it's a third process competing for the SQLite write lock (issue 2).
3. If it's **a method on the gateway**, the gateway must be running for indexing to happen. What about users who don't run the gateway?
4. The plugin can run without the gateway (per PRD config `gateway.enabled: false`). What happens to indexing then?

**Recommendation:**

Spell it out:

- **Default:** Indexer is an in-process worker thread inside the plugin AND a fallback path inside the gateway. Whichever process is alive does the work.
- **On clean shutdown** (`on_session_end`, plugin shutdown, gateway SIGTERM): indexer drains its queue before exit (with a 30s cap).
- **On crash:** Status is already on disk (`turns.index_status='pending'`); next process startup picks up where it left off.
- **No gateway?** Plugin's worker thread handles it solo. Catch-up indexing on plugin init: scan for any `pending` rows on startup.

Document this lifecycle explicitly in TDD §4.2 and add a "catch-up indexing on init" story to Phase 3.

## 3. Medium-Severity Issues (resolve during the affected phase)

### Issue 4: Daily Memory File Conflict with Existing Content

**Severity:** Medium
**Refs:** TDD §6.1, Plan Story 5.3.3

`~/.hermes/memories/YYYY-MM-DD.md` already exists today — the user has been writing entries there manually for months. The plan says the dreamer "preserves manually-edited content above an `<!-- AUTO-GENERATED BELOW -->` marker."

**Problems:**

1. Existing files don't have that marker. The first dreamer run will need a migration: detect existing files, preserve everything as manual content, append the auto-gen marker.
2. What if the user edits content **below** the marker between dreamer runs? Lost on next run.
3. What if the user deletes the marker? Auto-gen content overwrites everything.
4. No backup of the prior auto-gen section before regenerating.

**Recommendation:**

- Phase 5 includes a one-shot migration: each existing daily file gets the `<!-- AUTO-GENERATED BELOW -->` marker appended (empty section).
- Dreamer writes its section to a separate file: `~/.hermes/memories/.dreamer/YYYY-MM-DD.md`, and the daily file's auto-gen section is a transclusion / pointer. Decouples user content from auto-gen entirely.
- Alternative: keep both inline but maintain a `.bak` of the auto-gen section before regenerate.

Pick the first approach. Cleaner mental model.

### Issue 5: Hybrid Scorer Weight Math Doesn't Account for "Some Backends Returned Zero Results"

**Severity:** Medium
**Refs:** TDD §8.1

The hybrid scorer assumes all four backends produce candidate sets and merges them. But:

- FTS5 may return 0 candidates for a query the user phrased conceptually.
- Qdrant may return many candidates the FTS would never match.

If `W_FTS=0.30` but FTS contributed 0 candidates, the weight is "wasted" — the formula doesn't redistribute weight when a backend simply had no hits (only when it's unavailable).

**Problems:**

The scoring becomes "for candidates that FTS hit, multiply by FTS weight; otherwise multiply by 0 for that term." That means a candidate found only by Qdrant scores lower than it should, even when FTS is healthy but just didn't have a match.

**Recommendation:**

Use a max-or-average policy per candidate:

```python
# For each candidate, only include weights from backends that actually matched it
backends_matched = {b for b in backends if candidate in b.results}
total_weight = sum(W[b] for b in backends_matched)
relevance = sum(W[b] * scores[b] for b in backends_matched) / total_weight if total_weight else 0
```

This is the same redistribution logic as "backend unavailable," just per-candidate instead of system-wide. Document explicitly in TDD §8.

### Issue 6: Embedding Model Version in Collection Name — But Not in Chunk Hash

**Severity:** Medium
**Refs:** TDD §6.2 (`chunks` table), §6.3

The chunks table has `UNIQUE(text_hash, embed_model)`. The Qdrant collection name embeds the model version. Good. But:

**Problem:** The `text_hash` is computed from chunk text only. If we ever change the **chunking strategy** (size, overlap, boundary logic), same text could be re-chunked differently — but `text_hash` matches an old chunk, so insertion is blocked, and we never re-index with the new strategy.

**Recommendation:**

Add `chunker_version` column. The unique constraint becomes `UNIQUE(text_hash, embed_model, chunker_version)`. Default = "v1". Bump when chunking changes. Rebuild flow handles old vs. new chunker versions gracefully.

### Issue 7: `force_no_redact=true` Is a Bigger Footgun Than the Docs Suggest

**Severity:** Medium
**Refs:** TDD §7.1, PRD FR-015

The `force_no_redact` override lets the agent write a fact with secrets in it if the user explicitly asks. The docs say it's logged in audit_log.

**Problems:**

1. **Tool-using agents are bad at risk assessment.** An LLM that hits a "the user asked me to remember their api key" instruction might happily set `force_no_redact=true` because the user said so. The agent itself shouldn't have that authority.
2. The audit log is local-only. A compromised system can be tampered with.
3. There's no rate-limit or count on force_no_redact uses.

**Recommendation:**

- Either remove `force_no_redact` from the MVP entirely (start strict), OR
- Require a separate config flag at the Hermes process level (`plugins.hermes-local-memory.redaction.allow_override: false` by default). Even if the agent passes `force_no_redact=true`, redaction still runs unless the config flag is true. Override has to be a deliberate user act, not an agent decision.

I'd remove it entirely from MVP. Add it back in v0.3 if there's a real use case.

### Issue 8: `memory_recent_context` and `system_prompt_block` Will Re-Inject Stale Cached Content

**Severity:** Medium
**Refs:** TDD §15.2, Hermes core `_cached_system_prompt`

We use the same Hermes core `_cached_system_prompt` mechanism for the small `system_prompt_block` (pinned facts + active project) that just bit us with the narrative thread. Same cache invalidation issue applies:

**Problem:** If the user pins a new fact mid-session, `system_prompt_block` won't refresh to include it until next agent restart or compression. The agent's view of "pinned facts" goes stale.

**Recommendation:**

Don't try to put **anything dynamic** in `system_prompt_block`. Make it static metadata only ("Active memory provider: hermes-local. Use memory_query / memory_write."). Pinned facts should be served via `memory_recent_context` (which agents call explicitly) or `prefetch` (which returns fresh-each-turn). Cleaner design and avoids the cache problem.

### Issue 9: Dreamer LLM JSON Mode Assumption May Not Hold

**Severity:** Medium
**Refs:** TDD §10.2, §10.5

The dreamer prompts require strict JSON output. The config sets `json_mode: true`. The Qwen3.6-35B model running on LMS at `.105` may or may not honor LMS's structured-output / JSON mode reliably.

**Problems:**

1. If the model returns prose, the parser fails, the dream run fails or produces zero outputs.
2. No retry/repair loop documented.
3. No fallback to "ask model to fix its own JSON output."

**Recommendation:**

- Spike: validate Qwen3.6 + LMS json_mode produces clean JSON on the actual extraction prompts before Phase 5 starts. Document results.
- Add a parse-retry loop with up to 2 attempts. Second attempt includes the previous (malformed) output and asks the model to repair it.
- Surface parse failures in the dream report so the user sees what's broken.
- For MVP: if Qwen3.6 doesn't reliably JSON, fall back to constrained-output via `guidance` or `outlines` library on the gateway side.

### Issue 10: Auto-Promote Confidence Threshold of 0.8 Is Untested

**Severity:** Medium
**Refs:** PRD §11.4 (Promotion Levels), Plan Story 5.3.1

`auto_promote_confidence: 0.8` is a magic number with no basis in measurement. Whatever Qwen3.6-35B's calibration is, it might emit `confidence: 0.85` for fluff and `confidence: 0.5` for solid facts.

**Problems:**

1. Confidence is self-reported by the LLM; this is well-known to be unreliable.
2. No mechanism to recalibrate after we observe real data.
3. No "promotion review" tool visible to the user.

**Recommendation:**

- For MVP: don't auto-promote anything. ALL dreamer-extracted facts land with `status='candidate'`. User runs `memory review-candidates` (CLI) to promote in batches.
- Add to Plan Phase 5 as a small extra story.
- Track real confidence-vs-quality data over the first month; tune threshold in v0.3.

This avoids polluting durable memory with hallucinated facts in week one.

### Issue 11: Migration Script Doesn't Handle Holographic Schema Drift

**Severity:** Medium
**Refs:** TDD §16, Plan Story 6.1.1

The migration script assumes holographic schema as of today. If the Hermes team updates holographic upstream — new columns, renamed tables, dropped columns — our migration script breaks on the next pull.

**Problems:**

1. We pin nothing.
2. The migration is a one-time-then-discarded artifact in concept, but in practice users may run it across multiple Hermes upgrades.

**Recommendation:**

- Migration script reads holographic via SQLite introspection (`PRAGMA table_info`) and is resilient to extra columns (ignore them) or missing ones (use defaults).
- Pin the schema we assume in a version-check at the top: "Holographic schema version X.Y supported; found Z.W, behavior may vary."
- Add a `--dry-run` flag that shows what would migrate without writing.
- Document: "Re-run after future holographic upgrades only after reviewing release notes."

## 4. Lower-Severity Issues (track during build)

### Issue 12: Per-Session Thread File Cleanup Is Not Implemented

**Severity:** Low/Medium
**Refs:** PRD Q5, TDD §6.1

Thread files at `~/.hermes/SESSION-THREAD/{session_id}.md` will accumulate forever. Plan calls for 30-day retention rolled up by dreamer. Not implemented in MVP.

**Recommendation:** Add to Phase 5 as one small story: dreamer's nightly run also archives thread files older than 30 days into `memory/exports/thread-archive-YYYY-MM.tar.gz`. Files removed from active dir post-archive.

### Issue 13: No Plan for "Plugin Active But SQLite Locked" State

**Severity:** Low/Medium
**Refs:** TDD §6.4

If `memory.sqlite` is locked (e.g., another Hermes process is mid-write), the plugin's `initialize()` will fail and the agent's startup may be brittle.

**Recommendation:** `initialize()` retries SQLite connection with backoff for up to 10s. If still locked: degrade to in-memory mode (capture goes to JSONL only, no SQLite/Qdrant for this session) and log loudly. Better than refusing to start.

### Issue 14: Tool Schema Bloat — We're Adding 7 New Tools

**Severity:** Low
**Refs:** TDD §5.2

7 tool schemas (memory_query, memory_write, memory_update, memory_get_source, memory_recent_context, memory_dream_now, fact_feedback) — that's a lot of tool schema bytes in every agent call. Honcho has 5. Holographic has 2.

**Recommendation:**

- For agents using Hermes' default tool registration, all 7 schemas appear. This is fine.
- For very token-constrained models: future config option `plugins.hermes-local-memory.tool_subset: minimal` exposes only `memory_query` + `memory_write` and merges others into them via `action` enum (like fact_store's "action" pattern).
- Out of scope for MVP, but design `memory_query`'s mode enum to make this collapse easy later (already done — we have modes like `probe`, `related`, etc.).

### Issue 15: No Strategy for `tool_calls` That Contain Other Sessions' Source Refs

**Severity:** Low
**Refs:** TDD §7.3

If turn N contains a tool call whose result contains a `source_ref` to session M (e.g., the agent ran `memory_get_source('session:M#turn=3')`), capture stores this verbatim. Later, if session M is deleted/archived/migrated, that source_ref dangles.

**Recommendation:** Resolver returns `{kind: 'missing', source_ref: '...', reason: 'session archived'}` instead of an error. Agent can handle gracefully. No data integrity issue, just a UX nicety.

## 5. Things I Was Wrong About (or Hand-Waved)

These are honest second-look corrections to my own design:

### 5.1 "Hermes plugin loader picks it up — no separate install"

Not quite true. The plugin needs to be in `~/.hermes/hermes-agent/plugins/memory/hermes-local/`. If the user is on a clone elsewhere or a Hermes upgrade wipes the plugins dir during update, it could be lost. Add a sanity check on `is_available()` and clearer install docs.

### 5.2 "Both plugin and gateway share one SQLite database"

True, but "share" is doing a lot of work. They share the FILE; they don't share the CONNECTION. Connection pool semantics, WAL synchronization, and busy_timeout matter (see Issue 2).

### 5.3 "Dreamer prompts forbid hallucination"

A prompt directive forbidding hallucination is a wish, not a guarantee. Need validation post-extraction: every claimed fact must have a `source_ref` that actually resolves; reject items that don't.

### 5.4 "Qdrant 50/30/20 weights from OpenClaw enhanced-memory"

The OpenClaw weights are for 3 backends (Qdrant + QMD + LCM). We have 4 (Qdrant + FTS + Jaccard + HRR). Our default weights are different and aren't validated yet against real workloads. Plan for re-tuning after Sprint 5 + a week of dogfooding.

### 5.5 "Single tool surface eliminates agent drift"

True only at the Hermes provider level. The agent could still hit Hermes' built-in `memory` (the file-based tool that writes to `MEMORY.md`/`USER.md`). The `on_memory_write` hook mirrors those into facts, but there are now TWO write paths: `memory_write` AND `memory` (built-in). Document that explicitly so the agent can choose intentionally.
